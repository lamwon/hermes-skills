# Architecture Decision Meeting (架构决策会议)

## 适用场景

在项目正式编码前，需要对齐多方技术意见的场景：
- 脚手架搭建完成但核心架构存疑
- 前端/后端工程师对技术选型有不同倾向
- 产品需求与技术实现之间存在需要裁决的trade-off
- 有多达10-30项待定决策需要逐一讨论并记录

## 流程

### Step 1: 监理人起草议题清单

Hermes 作为监理人，阅读所有上下文后，输出一份 `docs/架构对接议程.md`：

```
# 架构对接议程

## A. 前端工程师提出的问题（来自前端评审报告）
列出前端工程师已明确提出的待定项

## B. 监理人发现的补充问题
从宏观角度补充遗漏的议题

## C. 需要总工程师裁决的架构决策
全局性的技术决策

## 输出要求
每个议题格式：
| # | 问题 | 选项A | 选项B | 建议方 |
```

### Step 2: 角色扮演会议

使用 `delegate_task` 同时扮演两个角色：

```python
delegate_task(
    goal="角色扮演架构对接会议：前端架构工程师 × 总工程师",
    context="...根据 docs/架构对接议程.md 逐项讨论...",
    toolsets=["file"]
)
```

讨论格式：
```
### A1. 议题标题
- 前端工程师意见：[理由]
- 总工程师裁决：[最终决定]
- 理由：[1-2句话]
```

### Step 3: 输出 ADR

输出 `docs/架构决策记录.md` (Architecture Decision Record)：

```
# 架构决策记录（ADR）

## A. 前端工程师提出的问题
## B. 监理人的补充问题
## C. 总工程师的架构决策

## 补充：数据文件与架构文档的字段差异修正
列出数据实际结构与文档描述不一致的项

## 总结
3-5句话概括最终架构的关键特征
```

## 实践案例

在"我在故我在"哲学网站项目中，通过此模式在30分钟内解决了23项决策（A1-A5 / B1-B6 / C1-C7 / D1-D5），包括：

| 决策 | 结果 |
|------|------|
| Tailwind v3 vs v4? | v3（稳定优先）|
| 哲学家单选/多选? | MVP单选，架构预留 |
| 分享页数据恢复? | URL search params |
| 概念库缓存? | localStorage 7天 |
| 佛道儒配图? | CSS意境背景，mood字段驱动 |
| 降级方案? | 三层兜底（重试→缓存→wisdom金句）|

## 关键成功要素

1. **议题清单要完整** — 遗漏的决策会在编码阶段反复返工
2. **两角色都要有真实意见** — 不能一边倒，要制造健康的意见张力
3. **总工程师做最终裁决** — 避免陷入无休止的讨论
4. **数据文件与文档的差异要记录** — 实际数据和架构描述往往不一致（字段名不同、结构不同），ADR 中专门记录修正表
5. **每条决策都要有理由** — 别说"采纳"，要说"采纳因为..."
