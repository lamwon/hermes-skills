# AI 提示词工程师 (Prompt Engineer)

为"我在故我在"哲学网站设计全套 DeepSeek V4 Flash 提示词系统。

## 三大核心提示词模板

### 1. 分析生成提示词 — 当前线上版

设计原则：费曼学习法五段式 + 硬性规则约束，AI 真正理解问题而非套模板。

```
## 任务
用户的问题是：「${question}」
请用${philosopher.name}（${philosopher.school}）的思想来回应。像一位懂哲学的朋友在聊天那样——先理解用户的处境，再用哲学家的理论去分析。

## 哲学家档案
姓名、流派、时代、核心思想、经典名言、代表作

${isMismatch ? "## 注意\n用户刻意选了一位看起来不相关的哲学家。请幽默地承认这种错位，然后仍然严格用TA的思想框架来回应。" : ""}

## 硬性规则
1. 先用自己的话复述用户问题的核心矛盾（不超过两句话）
2. 至少三次直接引用用户原文（用引号标注），进行哲学剖析
3. 建议必须来自这位哲学家特有的方法，不能给通用的"三步走"
4. 不要用"你以为你在问X，其实……"这种万能开头
5. 如果要提供其他视角，必须是另一位真实哲学家的具体观点
6. 用"你"称呼用户，口语化，像朋友聊天

## 输出结构（用【】标记每一段，一共5段）
【你的盲点】先指出用户问题里隐藏的预设，用原文点破（100字内）
【生活类比】把哲学家的核心思想类比到用户的生活场景，必须是一个具体画面（150字内）
【哲思解剖】引用哲学家原文逐句解析，关联到用户的具体困惑（200-300字）
【指向行动】给一个今天就能做的具体动作——不是"写下来贴墙上"，而是可执行的日常行动（100-150字）
【另一种可能】相反流派的观点，说明两种视角各有局限（100字内）
```

迭代历史：
- v1：通用费曼模板 → 内容不贴切用户问题
- v2：每段必须引用用户问题原文 + 固定结构【一句话说清楚】等 → 贴切了但不引用哲学家原文，且四段结构被用户反馈"太僵硬"
- v3-v4：加"铁律"要求每段引用原文 → 分析有深度但仍死板
- v5：统一用【你的盲点】【生活类比】【哲思解剖】【指向行动】【另一种可能】五段费曼结构，去掉"一句话说清楚"这种固定句式，改成先理解再分析。强调"指向行动"必须是今天能做的具体动作。
- v6（当前）：**即使 v5 结构合理，输出仍然僵硬**。用户反馈"所得答案还是太僵硬，且跟其他答案重复度较高，并没有用费曼学习法阐述剖析和指向行动"。

## 🔴 v5→v6 升级：解决"僵硬、重复、没费曼法"三大问题

### 问题诊断

用户用「身体不舒服」问不同哲学家，反馈核心痛点：
1. **僵硬** — 每段都像在套模板，不是真人对话
2. **重复** — 不同哲学家的回答结构完全一样（因为始终是同一套 mock 模板）
3. **没有费曼学习法阐述剖析** — "生活类比"段实际是哲学分析，不是日常画面
4. **指向行动形同虚设** — "把这句话抄下来贴在墙上"不是可执行动作

### 提示词级别改进（非代码）

#### 改进1：输出结构从"标题导航"改为"自然段落"

```markdown
## 输出结构（一共5段，直接用自然分段，不用固定标题）

不用死板的【你的盲点】【生活类比】等标题。直接写成流畅的自然段落，但内部逻辑是：

### 第1段：一句话点破盲区
- 用一句日常画面或具体场景开头（不是哲学概念）
- 指出用户问题里隐藏的一个未检视的预设
- 引用哲学家原文点破它

### 第2段：生活类比  
- 必须是一个具体的日常画面（去超市选酸奶、墙上的画歪了、做饭盐放多了、信号塔坏了）
- 把哲学家的核心思想嵌入到这个画面里
- 类比完再说「你的XX问题也一样」

### 第3段：原文剖析
- 引用哲学家原文，逐句解析，每解析一句就联系到用户具体场景
- 可以用"你解决过类似的事情吗？回忆一下——"来互动

### 第4段：可执行的行动指引  
- 不能是"写下来贴墙上""问自己是认知方式还是事物本身"
- 必须是今晚或明早就做的具体动作：
  — 打开备忘录列两栏对比
  — 设闹钟备注每天看10秒
  — 反过来做一件日常习惯
  — 用第三人称写一遍自己的困境
  
### 第5段：真实对立视角
- 必须是另一位真实存在的哲学家/流派
- 说明两种视角各自适合什么场景
- 不能"早晨做哲学家，下午做实干家"这种万能收尾
```

#### 改进2：风格轮换指令（让每次输出不一样）

```markdown
## 风格轮换（每次随机选一种角度切入）

第一段的切入角度随机选择以下一种：
A. 用一个日常场景开头（做菜、选酸奶、骑自行车）
B. 用"如果XX坐在你对面喝咖啡"开头
C. 用把问题「掰开看」的手术刀式开头
D. 用"你家客厅有一幅画歪了"这种比喻开头

第五段对立视角也要随机选以下风格之一：
A. 「实用主义者会说…但XX会反问…」
B. 「具体问题具体分析的思路是…但XX关注的是结构…」  
C. 「如果换个时代背景看这个问题…」
```

#### 改进3：反模式惩罚（明确禁止）

```markdown
## 惩罚规则
如果回答出现以下情况，必须重写：
- 使用"你以为你在问X，其实……"句式（old: "你以为是问题本身，其实是你的尺子歪了"）
- 5段中都引用同一句名言（旧问题：同一个quote在5段里重复出现）
- 给出"第一、第二、第三"的通用建议清单
- 在提供其他视角时使用"相反派"这种无具体所指的称呼（必须实名：边沁会说…/庄子会说…）
- 建议内容与其他哲学家的回答高度相似
```

#### 改进4：开头的"复述矛盾"改为"一句话场景"

旧 prompt 要求：先复述用户问题核心矛盾
新要求：用一句日常场景开头（不是复述，而是类比）
```
❌ "你说「身体不舒服」——你不是在问一个抽象问题，而是在一个具体处境里感到被困住了"
✅ "你问「身体不舒服」——这让我想起一个人站在一道门前拼命敲，却从来没想过门是不是锁着的"
```

## 🔴 关键教训：Mock 数据必须多模板随机

**这是用户反馈"太僵硬、重复度高"的核心原因。**

旧代码只有一个模板函数 `makeMockMatch` / `makeMockMismatch`，每次用固定句式填充变量 → 所有哲学家的回答结构完全一样。

**修复方案：定义 6+ 套不同风格的阐述模板，每次随机选一个。**

```typescript
const MOCK_FEYNMAN_VARIANTS = [
  // 模板组1：生活类比切入型（厨房做菜、敲门、信号塔）
  {
    oneliner: (q, n, s, qt) => `你问「${q}」——这让我想起一个场景：你站在一道门前，拼命敲，却从来没想过门是不是锁着的。...`,
    analogy: (q, n, s, qt) => `想象你在厨房里做饭，盐放多了。你不断地加水...`,
    deep: (q, n, s, qt, w) => `...`,
    advice: (q, n, s, qt) => `今晚就做一个动作：拿张纸，左边写「我担心的事」，右边写「这件事让我想到什么过去的经历」。...`,
    alternateView: (opp, n) => `...`,
  },
  // 模板组2：场景还原型（喝咖啡、手机信号塔）
  // 模板组3：问题重构型（鞋底薄了3毫米）
  // 模板组4：日常微行动型（墙上的歪画）
  // 模板组5：反差混搭型（超市选酸奶、学自行车）
  // 反差模式（不相关哲学家）专用2套
];

function makeMockMatch(...) {
  const variant = MOCK_FEYNMAN_VARIANTS[Math.floor(Math.random() * MOCK_FEYNMAN_VARIANTS.length)];
  return {
    oneliner: variant.oneliner(question, philosopherName, philosopherSchool, quote),
    ...
  };
}
```

关键原则：
- **每个模板组有不同的切入角度**（生活类比、场景还原、问题重构、日常行动、反差混搭）
- **"指向行动"必须是今天就能做的具体动作**（"设闹钟备注""拿张纸左右列两栏""反过来做习惯"），而不是"写下来贴墙上"
- **每套都包含真正的日常类比**（做饭、信号塔、歪画、学自行车），不是用哲学理论解释哲学理论

## 🔴 关键教训：parseAnalysis 不能 early return

```typescript
// ❌ 错误：匹配到第一个区块就 return，后面4段永远为空
for (const key of sections) {
  const val = extractSection(text, key);
  if (val) {
    return { ... , [fieldMap[key]]: val }; // ← return here!
  }
}

// ✅ 正确：遍历所有5个区块，各自独立提取
const result = { oneliner: "", analogy: "", deep: "", advice: "", alternateView: "" };
const sections = [
  { keys: ["你的盲点", "一句话说清楚"], field: "oneliner" },
  { keys: ["换个视角", "生活类比"], field: "analogy" },
  { keys: ["哲思解剖", "原文分析"], field: "deep" },
  { keys: ["指向行动", "生活建议"], field: "advice" },
  { keys: ["另一种可能", "换个角度"], field: "alternateView" },
];
for (const { keys, field } of sections) {
  for (const key of keys) {
    const val = extractSection(text, key);
    if (val) { result[field] = val; break; }
  }
}
// 只有全空时才 fallback 到段落切割
if (!result.oneliner && !result.analogy && ...) { return fallback; }
return result;
```

## 🔴 关键教训：前端必须调自己的 API 路由，不直连代理

旧架构：前端直接调 `http://127.0.0.1:15721/anthropic/v1/messages`（本地 cc-switch 代理）

问题：Vercel 部署后连不上 localhost，API 永远失败，永远走 mock 数据。

修复：**三层降级架构**
1. 主路径：`fetch("/api/analyze")` — 自己的 Next.js 路由，SSE 流式接收，在 Vercel 上也能工作
2. 降级1：`DEEPSEEK_PROXY_URL`（本地 cc-switch 代理），直接 Anthropic 格式调用
3. 降级2：mock 数据（6 套随机模板）

环境变量配置：
```
DEEPSEEK_PROXY_URL=http://127.0.0.1:15721/anthropic/v1/messages  # 开发环境
DEEPSEEK_API_KEY=sk-xxx  # 生产环境（Vercel）
```

## 🟡 关键教训：不同文件的提示词必须统一

旧代码中 `lib/api.ts` 的 hard rules 说"不要使用固定模板"，而 `route.ts` 的 prompt 却要求输出固定格式【一句话】【类比】【原文】【建议】——两边自相矛盾，AI 不知道听谁的。

修复：统一两个文件的提示词结构、段名、硬性规则，保持一致。

## 🟡 关键教训：输出段标记用 【】 比自由格式好

原来试图让 AI "自由组织结构，整体是一篇流畅文章" → parseAnalysis 无法提取分段内容。改用明确的 `【你的盲点】` 等标记后，后端可以准确提取。

每个段标记对应两个备选名称（新旧兼容）：
- `oneliner` ← 「你的盲点」/「一句话说清楚」
- `analogy` ← 「换个视角」/「生活类比」
- `deep` ← 「哲思解剖」/「原文分析」
- `advice` ← 「指向行动」/「生活建议」
- `alternateView` ← 「另一种可能」/「换个角度」
